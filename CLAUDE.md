# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

This repository contains a single self-contained file, `index.html`, implementing a one-page marketing/landing site for a fictional Singapore healthcare clinic ("Wellview Clinic"). There is no build step, no package manager, no bundler, and no test suite — all HTML, CSS (inline `<style>` in the `<head>`), and JavaScript (inline `<script>` before `</body>`) live in that one file. `robots.txt` and `sitemap.xml` at the repo root support crawlability; there's no other tooling.

## Commands

There is no build, lint, or test tooling in this repo. To preview changes, open `index.html` directly in a browser (double-click it, or `Start-Process index.html` on Windows) — there is no dev server to start.

## Architecture

Everything is in `index.html`, organized top-to-bottom as: sticky navbar → hero → services → "Meet Our Doctors" → testimonials → "Your First Visit" (journey + lead magnet, `#first-visit`) → enquiry form (`#contact`) → footer. Each section is marked with an HTML comment banner (e.g. `<!-- ================= HERO ================= -->`) — use these to jump to the right region instead of searching class names.

- **Styling**: all CSS lives in one `<style>` block, driven by CSS custom properties defined once in `:root` (colors, radius, shadow, max-width, `--font-display`). Change the palette/spacing by editing those variables rather than individual rules. Section-specific styles are grouped under comment headers (`/* ---------- Hero ---------- */`, etc.) in the same order the sections appear in the HTML.
- **Typography**: two-role pairing — `Fraunces` (a warm display serif, `var(--font-display)`) for `h1`/`.section-heading h2`/checklist and journey-number headings, and `Poppins` for everything else (body copy, nav, buttons, labels). Both are loaded from the same Google Fonts `<link>` in `<head>`. Keep this pairing rather than reverting to a single font — see [Site context for design/marketing skills](#site-context-for-designmarketing-skills) below for why.
- **Scroll behavior**: nav links are plain in-page anchors (`href="#services"`) relying on `html { scroll-behavior: smooth }` — no JS-driven scrolling.
- **Scroll-triggered animation**: any element with the `.fade-in` class is faded/slid in via a single `IntersectionObserver` in the inline script; it falls back to revealing everything immediately if `IntersectionObserver` is unsupported. Card grids (`.services-grid`, `.doctors-grid`, `.testimonials-grid`, `.journey-grid`) get a staggered `transition-delay` via `nth-child` CSS rules — add a matching `nth-child(n)` rule if a grid grows past 4 items.
- **Mobile nav**: the hamburger button toggles an `.open` class on `#nav-links` and updates `aria-expanded`; there's no separate mobile markup, just a CSS breakpoint at `859px`/`860px`.
- **"Your First Visit" (`#first-visit`)**: a real content sequence (`.journey-grid`, an `<ol>` since the 3 steps are genuinely ordered) plus the site's lead magnet — an ungated, on-page "First Visit Checklist" (`.checklist-card`). The Download/Print/"email it to me" buttons are wired up in the inline script: Download builds a `Blob` and triggers a real file save (no backend needed), Print calls `window.print()` against a `@media print` rule scoped to `#checklist-print`, and the email link just pre-fills the enquiry form's message field — none of them pretend to call a server that doesn't exist.
- **Enquiry form (`#enquiry-form`)**: fully client-side. On submit it prevents the default action, validates name/email/phone with regexes, toggles `aria-invalid` + a per-field `.error-message` span, and on success logs a plain data object to the console and shows a `.form-status.success` banner. The block comment inside the submit handler marks exactly where a real `fetch('/api/appointments', ...)` POST would replace the `console.log` — there is no backend.
- **Images**: person photos (hero, doctor cards, testimonial avatars) are hotlinked directly from `images.unsplash.com/photo-<id>?...` URLs with explicit `w=` size params sized for their display context (800px for hero/doctor cards, 300px for testimonial thumbnails) — the page requires network access to render these; there are no local image assets. All except the hero photo use `loading="lazy"` + explicit `width`/`height`; the hero photo is `fetchpriority="high"` since it's the LCP element.
- **SEO**: `<head>` carries a meta description, canonical link, Open Graph/Twitter tags, and a `MedicalClinic` JSON-LD block (name, address, hours, specialties) built from the real contact details in the Contact section/footer — keep those three in sync if they ever change. `robots.txt`/`sitemap.xml` at the repo root point at the GitHub Pages URL.
- **Icons**: small inline-with-text icons (trust strip, contact info) are hand-written inline SVGs (`stroke="currentColor"`, stroke-width 1.8) rather than emoji, for consistent rendering at small sizes. The larger circular service-icon badges intentionally keep emoji — that's a deliberate style choice, not an oversight, since they read as an illustrated badge rather than inline text.

## Conventions worth preserving

- Keep new sections consistent with the existing pattern: a `.section-heading` block (`eyebrow` + `h2` + intro `p`), a responsive `grid-template-columns: repeat(auto-fit, minmax(...))` card grid, and `.fade-in` on major blocks for the scroll-reveal effect.
- Form fields follow a fixed structure: `.field > label + input/textarea + span.error-message`, with `id="{name}-error"` wired via `aria-describedby` on the input — replicate this for any new form field so the existing `setFieldError()`/`validateForm()` JS logic can be extended consistently.

## Site context for design/marketing skills

This project has the `frontend-design`, `ui-ux-pro-max`, `seo-audit`, and `lead-magnets` skills installed (`.claude/skills/`, tracked via `skills-lock.json`). When any of them run in this repo, ground their generic guidance in these actual site facts instead of inventing a brief from scratch:

- **Brand**: Wellview Clinic, a general/family healthcare clinic in Singapore. Voice is warm, reassuring, and professional — never clinical-cold or salesy. Audience is Singapore families and working adults choosing a trusted local clinic for themselves and their kids, so copy and design choices should read as approachable and credible, not corporate-generic.
- **Palette** (from `:root` in `index.html` — treat these as the source of truth, don't invent a new palette): primary teal `#0d6e7d` / `#094f5a`, secondary sky-blue `#3fa9c9`, accent coral `#ff8a65` / `#f4693a`, background white `#ffffff` / off-white `#f4fafb`, text `#1f2d33` / `#5a6c72`.
- **Typography**: `Fraunces` (display headings) paired with `Poppins` (body/UI) — see the Architecture section above. Any further type-pairing recommendation should either work with this pairing or propose a deliberate, justified replacement — not silently swap it.
- **Real content to anchor to**: services are general consultation, pediatric care, diagnostics & lab, dental care; the four doctor profiles and testimonials in `index.html`; location 21 Orchard Boulevard #03-08, Singapore 248649; hours Mon–Sat 8am–8pm, Sun 9am–1pm.
- **SEO**: this is a fictional local-business site — frame `seo-audit` findings around local/healthcare SEO (Google Business Profile, NAP consistency, service-area keywords like "family clinic Singapore" / "pediatric clinic Orchard") rather than generic e-commerce/SaaS SEO advice. Meta tags, canonical, OG/Twitter tags, `MedicalClinic` JSON-LD, `robots.txt`, and `sitemap.xml` are already in place (see Architecture above) — extend them rather than re-adding from scratch. There's no real backend or analytics wired up, so flag but don't assume server-side fixes are actionable without a backend.
- **Lead magnets**: the site already has one, live at `#first-visit` — an ungated "First Visit Checklist" with a working client-side download (`Blob` + triggered save) and print flow, plus a link that pre-fills the `#contact` enquiry form. Any further lead magnet ideas should follow the same pattern (ungated, genuinely functional client-side, no fake "we emailed it to you" claims) since there's no email platform or backend connected.
- **`ui-ux-pro-max` security note**: this skill was flagged High Risk by the Gen scanner at install time (Socket/Snyk were clean). Its data/scripts haven't been fully audited — review `.claude/skills/ui-ux-pro-max/scripts/` before letting it execute anything beyond reading its reference CSVs.
